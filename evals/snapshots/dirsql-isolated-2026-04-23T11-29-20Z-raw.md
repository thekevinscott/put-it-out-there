## Scope of the existing release process

The release machinery in `.github/workflows/` is 7 files + 4 Python scripts doing roughly four distinct jobs. Reading them end-to-end:

- **`release.yml`** — autogenerated by `cargo-dist`, triggers on `v*` tags, builds `.tar.xz` / `.zip` archives for 5 targets, uploads to a GitHub Release. Pure binary-archive distribution; not a registry publisher.
- **`patch-release.yml`** — the actual orchestrator (confusingly named). Runs on cron (02:00 UTC), push-to-main with "immediate" strategy, or workflow_dispatch. Decides `should_release` + `bump_type` via a shell script, detects changed SDKs via `scripts/release/resolve_publish_targets.py`, then calls `publish.yml` as a reusable workflow. File-named `patch-release.yml` deliberately — the name is pinned in crates.io and npm trusted-publisher JWT claims (comment at `patch-release.yml:1-4`).
- **`publish.yml`** — tag creation (`compute_version.py`), maturin wheel build matrix across 5 targets with a `cargo build --bin dirsql --features cli` stage-into-wheel dance (`publish.yml:158-188`), PyPI trusted-publisher upload with `skip-existing: true`, crates.io publish guarded by `check_published.py`, GitHub Release creation, and a tag-rollback job that fires when both pypi + crates fail.
- **`publish-npm.yml`** — triggered by `workflow_run` on `release.yml` completion (the cargo-dist one). Runs a `napi build` matrix across the same 5 targets, downloads cargo-dist's archives from the GitHub Release, synthesizes per-platform `@dirsql/cli-*` and `@dirsql/lib-*` packages via `tools/buildPlatforms.ts` / `tools/buildLibPlatforms.ts` / `syncVersion.ts`, publishes each, then publishes the top-level `dirsql` with `optionalDependencies` pinning. Uses a classic `NPM_TOKEN`, not OIDC.
- **`scripts/release/`** — ~4 Python scripts (`compute_version`, `check_published`, `resolve_publish_targets`) with their own tests and `pyproject.toml`. These are the domain logic behind the YAML.

That's the "day of work." Most of it is the kind of thing piot is designed to absorb: change detection, version math, registry pre-checks, idempotent publish, the napi platform-package dance, tag + Release creation.

## What piot already covers for dirsql (green)

Evidence cited against `guide/handoffs/polyglot-rust.html` and `guide/concepts.html`:

1. **The three registries dirsql targets are exactly piot's three** — crates.io, PyPI, npm. The handoff guide's sample config is literally named `dirsql-rust` / `dirsql-py` / `dirsql-napi`; this shape is piot's design case.
2. **Cascade + `depends_on` replaces `resolve_publish_targets.py`** — `paths = ["packages/rust/**", "Cargo.toml", "Cargo.lock"]` plus `depends_on = ["dirsql-rust"]` on the wrappers gets you the same "Rust change => all three ship" behavior that `resolve_publish_targets.py` currently computes from git diff.
3. **Idempotent `isPublished` pre-check replaces `check_published.py`** — concepts page explicitly: *"Every handler's first move is isPublished — check the registry for the target version. Already there? Skip cleanly."*
4. **OIDC trusted publishing on all three registries** — dirsql already uses OIDC for PyPI (`pypa/gh-action-pypi-publish`) and crates.io (`crates-io-auth-action`), so piot's happy path is compatible. npm would migrate from today's long-lived `NPM_TOKEN` (`publish-npm.yml:163`) to OIDC — genuine improvement.
5. **napi platform-package family publish is a first-class piot feature** — `build = "napi"` with `targets = [...]` does what `buildPlatforms.ts` / `buildLibPlatforms.ts` / `syncVersion.ts` do today by hand.
6. **Scheduled cadence is supported** — `putitoutthere init --cadence scheduled` exists (CLI reference), so the daily-cron model survives.
7. **Trailer grammar maps cleanly onto today's knobs** — `release: skip` replaces dirsql's `[no-release]` token (`patch-release.yml:82-86`); `release: minor` replaces the workflow_dispatch dropdown.
8. **GitHub Action `thekevinscott/put-it-out-there@v0`** means no Node install step in CI even though the CLI is npm-distributed.

## What piot deliberately doesn't cover (but dirsql needs)

All compose-style; no real friction:

- **Binary-archive uploads to GitHub Releases** — today's `release.yml` (cargo-dist). Concepts: *"Standalone binary archive uploads to GitHub Releases... that is cargo-dist's and goreleaser's territory; compose with them."* Keep cargo-dist as-is.
- **Staging the CLI binary into the wheel** — the `cargo build --bin dirsql` + `cp` dance at `publish.yml:172-188`. Handoff guide addresses this directly: *"piot doesn't have a pre-build hook for this yet; the staging step stays in your build job."* OK; the wheel is what piot publishes, staging stays in the user-owned `build` job.
- **Auto tag-rollback on partial failure** — `publish.yml:350-374`. Piot deliberately drops this; the completeness check pre-publish is the replacement. Behavior loss is acceptable (the existing rollback is already conditional and has a known bug path for TS-only releases, per the comment).

## Blockers that must be addressed before dirsql can adopt piot

These are the real "highlight and fix" items:

### 1. Workflow-filename collision with cargo-dist (hard blocker)

Piot's `init` scaffolds `.github/workflows/release.yml` (Getting Started + CLI reference + Action reference all hardcode this name). dirsql's `release.yml` is already taken — it's the cargo-dist workflow — and the trust-policy filename is pinned in the OIDC JWT at crates.io and npm. Today dirsql worked around this by naming the orchestrator `patch-release.yml`; the comment at `patch-release.yml:1-4` is explicit about the consequence of a rename.

Piot docs currently have **no guidance** for installing alongside a pre-existing `release.yml`. What's needed, in order of preference:

- A `workflow = "piot.yml"` knob in `putitoutthere.toml` that the scaffolder, the Action docs, and `doctor`'s trust-policy guidance all honor; OR
- A `putitoutthere init --workflow-filename <name>` flag; OR
- A documented "composing with cargo-dist" recipe that walks through the filename choice and trust-policy implications.

Without one of those, adopting piot forces a trust-policy re-registration on two registries as a precondition, which partially defeats "avoid the day of work."

### 2. Workspace-inherited Cargo version isn't addressed (hard blocker)

dirsql's crates use `version.workspace = true` (`packages/rust/Cargo.toml:4`) and the literal `version = "0.1.0"` lives at the root `Cargo.toml:6`. Today `publish.yml:282-288` `sed`s the root with a comment explaining exactly why.

The handoff config declares `[[package]] name = "dirsql-rust", path = "packages/rust"`. It's not documented whether piot's crates handler:
- edits `packages/rust/Cargo.toml` (wrong — that manifest has no literal version), or
- walks up to find the workspace root, or
- needs a `manifest_path = "Cargo.toml"` / `workspace_root = ".."` override.

This is the single concrete question that would decide whether `putitoutthere publish` actually produces a correctly-versioned crate on the first run. The config reference should answer it.

### 3. Tag-scheme migration is called out but not solved

Handoff guide, "Gotchas": *"piot tags each package independently as `{name}-v{version}`... If your existing setup used a single shared `v0.3.1` tag across all three, consumers reading tags... need to update."*

Concrete fallout in this repo:
- `publish.yml:67-68` hard-codes `grep -E '^v[0-9]'` to find the latest tag for version math. Gone with piot, but any other tool still doing this breaks.
- `publish-npm.yml:34` triggers on `startsWith(github.event.workflow_run.head_branch, 'v')` — likewise.
- Any install docs / release pages / scripts that reference `v{version}` URLs.

Piot either needs a `tag_template = "v{version}"` escape hatch (lockstep shared-tag mode) or dirsql needs to accept three parallel tag streams and audit the repo for `v*` consumers. Handoff guide should recommend one.

### 4. Lockstep semver across the three packages isn't documented

Related to #3 but separate: dirsql today ships `dirsql` crate, `dirsql` wheel, `dirsql` npm package all at the same version from the workspace root. With three independent `[[package]]` entries, piot cascades on file-glob membership — a Python-only change bumps only `dirsql-py`, diverging the three versions. That's arguably the right behavior, but it's a real semantic change the handoff guide doesn't discuss. If lockstep is a requirement, piot needs a `lockstep = true` mode or a clearly-documented "make all three packages own every path" pattern. Today neither exists in the docs.

### 5. Install path is npm-only (soft blocker for the general library)

CLI reference: *"putitoutthere is the CLI shipped with the npm package."* Getting Started only shows `npx putitoutthere init`. For dirsql this happens to be fine (Node is already present), but a Rust-only or Python-only consumer of piot hits a surprising npm dependency just to scaffold. Given piot's stated audience is "polyglot Rust/PyPI/npm" projects, npm is almost always present — but the docs should state that explicitly, or ship `pip install putitoutthere-cli` / `cargo install putitoutthere` wrappers.

Not a dirsql blocker. Worth flagging for the library's broader appeal.

---

## Bottom line for dirsql adoption

The fit is **real and close** — piot's polyglot-rust handoff is clearly authored with this repo in mind. If blockers #1 and #2 are resolved (workflow-filename knob + workspace-root manifest handling), a dirsql migration is a half-day task, not a day: delete `patch-release.yml` + `publish.yml` + `scripts/release/`, keep `release.yml` (cargo-dist) + delete `publish-npm.yml`, add `putitoutthere.toml` (≈30 lines from the handoff's example), re-register trust policies on three registries with the new filename, switch npm to OIDC. Net: ~650 lines of YAML + 4 Python scripts gone.

Without #1 and #2 resolved, adoption stalls the first time `putitoutthere publish` tries to edit the wrong Cargo.toml or the scaffolder clobbers cargo-dist's `release.yml`. Those two are the concrete things to fix in piot before pitching it to dirsql.
