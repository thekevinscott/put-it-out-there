## Repo conventions worth internalising before editing

1. **`notes/design-commitments.md` is normative.** Any proposed feature has to survive the decision test there ("would release-please, cargo-dist, or changesets also want this? if yes, it's a non-goal"). The whole #172 refactor was driven by that file; future eval-driven proposals should be checked against it first.
2. **Bundle discipline.** Any change to `src/**` must be followed by `pnpm run build:action`, and the resulting `dist-action/index.js` committed alongside. CI has a check named `dist-action/ is up to date` that fails loudly otherwise. Merge conflicts in this file on polyglot PRs are normal — resolve by taking one side and rebuilding rather than trying to reconcile the minified output.
3. **Coverage threshold is 95% across all four axes** (lines/statements/branches/functions) — see `vitest.config.ts`. New code often exposes defensive branches the tests can't reasonably hit; prefer simplifying the code (optional chaining, narrower types) over adding `/* v8 ignore next */` comments. The `hasDynamicVersion` rework in `be9448f` is a good reference.
4. **Commit message style**: `<type>(<scope>): <subject> (#<issue>)`. Types include `fix`, `feat`, `perf`, `refactor`, `test`, `docs`, `chore`. Keep the body focused on *why*, not *what*.
5. **Branch naming**: `claude/issue-<N>-<slug>`. PRs are squash-merged with auto-merge after CI goes green.
6. **The `thekevinbot` account** is what these PRs are authored as (GitHub MCP integration). That's fine — humans review and merge via `thekevinscott`.

## Useful patterns from this session

- **Parallel worktree agents**: for 4 independent issues touching disjoint files (crates.ts, npm-platform.ts, pypi.ts, PLAN_GAPS.md), spawning 4 `Agent` calls with `isolation: "worktree"` worked well. Each got its own checkout, pushed its own branch, and reported back. Total wall time ~6 minutes for all four vs. ~20 if run serially. Don't try this pattern when branches touch overlapping files or depend on each other.
- **Webhook subscription for auto-merge PRs**: `mcp__github__subscribe_pr_activity` + `enable_pr_auto_merge` together means you can fire-and-forget; CI failures arrive as `<github-webhook-activity>` messages for you to triage.
- **Coverage drop below 95% after a PR**: don't reach for `/* v8 ignore next */`. First try to simplify the code to remove unreachable defensive branches (`!parsed || typeof parsed !== 'object'` after a `try` that would have thrown — that kind of thing). That's a more honest fix and usually what the threshold was designed to surface.
